# ОБВАЛ — Технический план: воксельное разрушение (VS0)

> Живой документ. Дата: 2026-06-23. Родительский: `GDD_Obval.md`.
> Движок: UE **5.8**, модуль `NEXTGENONE` (C++). Цель: **Steam Deck 800p / 40 fps (25 мс/кадр)**.
> Назначение: спроектировать и построить **Vertical Slice 0** — прототип, доказывающий, что воксельное
> разрушение + структурная целостность (а) фаново и (б) влезает в кадровый бюджет Дека. **Гейт перед любым контентом.**

---

## 0. Рамки (что VS0 доказывает, а что — НЕ делает)

**Гибрид (повтор из GDD):** воксельные — только **разрушаемые структуры** (руинные секции/объекты). Ландшафт/фон/реквизит — обычная геометрия. Воксельную цену платим только там, где есть разрушение.

VS0 **доказывает:**
- меширование вокселей в realtime влезает в бюджет;
- разрушение (удаление вокселей → ремеш) не даёт хитчей;
- структурная целостность (отрыв → падение) работает и читается;
- движущийся фронт **Каскада** + заморозка пройденных секций держат активную зону ограниченной;
- всё это вместе ≤ 25 мс на Дек-классе.

VS0 **НЕ делает:** контент, PCG-генерацию разрезов, мету, арт, звук, врагов, сетку. Один рукотворный тестовый разрез.

## 1. Ключевые тех-решения (с дефолтом и спайком)

| Решение | Дефолт для VS0 | Альтернатива / спайк |
|---|---|---|
| Размер вокселя | **0.25 м** (стартовая точка) | 0.1–0.2 м — измерить стоимость; крупнее = дешевле |
| Размер чанка | **16³ вокселей** (=4 м куб при 0.25) | 32³ — меньше чанков/дроуколов, но дороже локальный ремеш |
| Хранение воксельных данных | плоский `TArray<uint8>` на чанк (material id, 0=пусто) | палитра/RLE-сжатие — позже, под память |
| Меш-компонент | **ProceduralMeshComponent** (ноль зависимостей, для спайка) | **RealtimeMeshComponent** (TriAxis) для прод — threaded, LOD; проверить 5.8 |
| Меширование | **greedy meshing**, async на воркере | naive face-culling (проще, больше треугольников) |
| Коллизия | упрощённая (box/convex по solid-кластерам), async-cook, только активные чанки | complex из меша — измерить стоимость cook |
| Структурная целостность | **бинарная связность** (flood-fill от якорей; оторвано → падает) | strain/load-модель (распределение веса) — Фаза 2 |
| Оторванный кусок | спавн динамического тела Chaos (convex collision) | мердж/деспавн осевшего дебриса |
| Воксельный плагин | **своя фокусная система на 5.8** | спайк: Voxel Plugin 2 на ветке 5.7 — сравнить трудозатраты/перф |
| Освещение | **Lumen off**, distance-field тени, без bounce-GI | — (решено в GDD) |

> Принцип: VS0 — на минимуме зависимостей (ProceduralMesh, своя система), чтобы быстро дойти до замера. Миграция на RealtimeMeshComponent — только если ProceduralMesh не тянет.

## 2. Архитектура данных

```
FVoxel            : uint8 materialId (0 = empty)
FVoxelChunk       : TArray<uint8> Voxels[16*16*16]; FIntVector ChunkCoord; bool bDirty;
                    (+ кэш меша, флаг наличия коллизии)
AVoxelStructure   : actor — одна разрушаемая структура/секция.
                    TMap<FIntVector, FVoxelChunk> Chunks;
                    список якорей (anchor-вокселей/чанков);
                    владеет меш-компонентами (по чанку или по секции).
UVoxelMesher      : статeless — greedy meshing чанка с учётом соседей; вызывается async.
FStructuralSolver : связность от якорей; поиск оторванных компонент.
UVoxelDamage      : применение урона (форма → удаление вокселей → пометка dirty-чанков).
AVoxelDebris      : динамическое тело Chaos для упавшего куска.
```

- **Индексация вокселя в чанке:** `idx = x + y*16 + z*256`.
- **Координаты:** мир → чанк (`FloorDiv`) + локальный воксель (`Mod`). Соседние чанки нужны для корректного face-culling на границах.
- **Material id** в VS0: 2–3 материала (камень / укреплённый / «слабый»). Хватает для палитры меша и для разной прочности.

## 3. Меширование (greedy)

- Генерируем **только видимые грани**: грань между solid и empty (или границей чанка с пустым соседом).
- **Greedy:** объединяем компланарные смежные грани одного материала в крупные квады по 6 направлениям → на порядок меньше треугольников, чем naive.
- **Async:** меш строится в `UE::Tasks::Launch` (или `FAsyncTask`) на воркере; результат (вершины/индексы/нормали/UV) применяется на game thread (`CreateMeshSection`/обновление). **Никогда не блокируем game thread.**
- **Double-buffer:** пока строится новая версия секции чанка, старая рендерится; свап атомарный на game thread.
- **Грязные чанки:** разрушение помечает затронутые чанки `bDirty`; в конце кадра собираем dirty-список и пускаем ремеш пачкой (с бюджетом N чанков/кадр).
- Один меш-компонент **на чанк** (проще инвалидация) — следить за числом дроуколов; если много, группировать по секции.

---

## 4. Коллизия

- Динамический ремеш + коллизия = главный перф-риск (cook на лету).
- **Стратегия VS0:** упрощённая коллизия — не complex-из-меша, а box/convex по кластерам solid-вокселей чанка; **async cook** (Chaos), **только для активных чанков** в радиусе взаимодействия игрока.
- Замороженные/дальние секции — без обновления коллизии (или статическая запечённая).
- Замерить: стоимость cook на чанк, число cook/сек при активном сносе. Если дорого — огрублять (один convex на чанк) или отложенная коллизия.

---

## 5. Разрушение (tools → voxels)

Поток: **инструмент → форма урона → удаление вокселей → dirty-чанки → ремеш (+ коллизия) → проверка целостности.**

- **Бур/кирка:** малая сфера/капсула, точечное удаление, **без структурного шока**.
- **Заряд:** большая сфера, быстрое удаление + **высокий структурный шок** (триггерит проверку целостности шире, может ронять).
- **Опора/балка:** обратная операция — **добавление** вокселей материала-опоры (контр-глагол; держит путь отхода).
- «Структурный шок» = вес/срочность запуска `FStructuralSolver` на затронутом регионе.

---

## 6. Структурная целостность (сигнатурная механика)

**MVP — бинарная связность:**
1. Якоря = «земля/коренная порода» или назначенные anchor-воксели.
2. При разрушении запускаем **flood-fill / BFS по связным solid-вокселям** в затронутом регионе (не по всему миру).
3. Компоненты, не достигающие ни одного якоря → **оторваны** → конвертируем в `AVoxelDebris` (динамическое тело, падает).
4. Event-driven, не каждый кадр; бюджет на число вокселей в проверке/кадр; крупные проверки разбиваем по кадрам.

**Дебрис → физика:**
- Меш оторванного куска (тот же greedy), коллизия = **convex hull** (дёшево), тело Chaos.
- **Кап на одновременные тела** (Дек CPU): напр. ≤ 50; мелочь мерджим; осевшее деспавним по таймеру (или превращаем обратно в статичные воксели).

**Фаза 2 (после VS0):** strain/load — распределение веса вниз по опорам, порог напряжения → разрушение перегруженного. Даёт «реалистичное» прогрессивное обрушение. В VS0 не нужно: бинарная связность уже даёт «убрал опору → упало».

---

## 7. Вертикальные секции + Каскад

- Разрез = вертикальный стек **секций** (в VS0 — 2–3 рукотворных).
- **Стриминг:** полностью симулируем только N ближних секций; пройденные сверху → **заморозка** (рендер статичного меша, без обновления сим/коллизии) либо выгрузка (но путь отхода держим живым).
- **Каскад** = движущийся по Z «фронт активации»: достигая секции, триггерит её обрушение (ослабление якорей секции → связность роняет всё) и выгружает секцию позади себя. Дёшево и драматично — **не полный сим везде**.

## 8. Бюджет производительности (Steam Deck)

**Цель: 25 мс/кадр (40 fps @ 800p).** Ориентир раскладки:
- GPU рендер: ~14–16 мс (Lumen off; DF-тени; контролируем дроуколы и овердро).
- Game thread: ~6–8 мс (логика, свап мешей, спавн дебриса).
- Воркеры (вне критического пути): меширование, `FStructuralSolver`, cook коллизии — параллельно.

**Лимиты (стартовые, валидировать):**
- Активные симулируемые чанки рядом с игроком: **≤ ~30–40**.
- Треугольников после greedy в активной зоне: десятки тысяч, не миллионы.
- Одновременные динамические тела дебриса: **≤ 50**.
- Ремеш-бюджет: **N чанков/кадр** (остальное в очередь).
- Память: воксельные данные дёшевы (байты); основная цена — буферы мешей → выгружать дальние секции.

**CPU — узкое место (Zen2 4 ядра/8 потоков):** вся тяжёлая воксель-работа на джоб-систему; game thread только применяет результаты. AI/прочее держать лёгким.

## 10. Порядок сборки VS0 (каждый шаг — чекпоинт)

1. **Данные + один статичный чанк** замеширен (greedy) и отрендерен. *Доказывает: меширование + рендер.*
2. **Мульти-чанк структура** + face-culling на границах чанков.
3. **Async-меширование** (вне game thread) + ремеш по dirty при правке.
4. **Разрушение** (удаление вокселей формами инструментов) → ремеш затронутых чанков.
5. **Коллизия** (упрощённая, async) + взаимодействие игрока.
6. **Структурная целостность** (связность) → оторвано → тело Chaos.
7. **Каскад-фронт** + заморозка/выгрузка вертикальных секций.
8. **Перф-пасс + замер на Деке** против 25 мс.

> Если к шагам 4–5 меширование/разрушение не лезут в бюджет — **пересматриваем тех-подход, не дизайн** (спайк VP2-на-5.7 / другой меш-компонент / крупнее воксель).

## 11. Спайки и открытые тех-вопросы

- **Меш-компонент:** ProceduralMesh vs RealtimeMeshComponent vs DynamicMesh — построить один чанк на каждом, сравнить перф/удобство/5.8-совместимость.
- **VP2 на 5.7** как альтернатива всей своей системе — короткий спайк, оценить трудозатраты vs контроль.
- **Коллизия:** complex-cook vs упрощённая — измерить стоимость при активном сносе.
- **Размер вокселя/чанка** — измерить перф на 0.1 / 0.2 / 0.25 м и 16³ / 32³.
- **Nanite-из-runtime-вокселей** (подход Victor Careil) — research-only, для MVP скорее оверкилл.

## 12. Критерий успеха VS0 (pass/fail гейт)

**PASS:** на Дек-классе (800p, Deck device profile) при активном сносе тестовой структуры с падением оторванных кусков и проходом Каскад-фронта **держится ≤ 25 мс/кадр** без хитчей > ~5 мс, и разрушение **субъективно фаново** (тактильно, читаемо, причинно).

**FAIL → ветки:** крупнее воксель / агрессивнее стриминг / упростить коллизию / сменить меш-компонент / спайк VP2-на-5.7. Менять **тех**, дизайн не трогаем.
